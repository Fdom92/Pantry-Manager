## §1 Streak counter

### Definition (per brainstorm Q6 = C)

- Counter increments when user logs **any pantry mutation** on a calendar day.
  - Pantry mutation = a persisted `PantryEvent` of type `ADD | CONSUME | EDIT | DELETE`.
  - First mutation of the day triggers the increment. Subsequent same-day mutations have no further effect.
- "Day" = local timezone, midnight-to-midnight.
- **Grace day rule**:
  - User has 1 unused grace day per rolling 7-day window.
  - If user misses 1 day → grace burned, counter unchanged.
  - If user misses 2 consecutive days with no grace available → counter resets to 0.
  - Grace regenerates exactly 7 days after the date it was consumed (sliding window).

### Data model

New persisted PouchDB doc (singleton):

```ts
// core/models/retention/streak.model.ts
import type { BaseDoc } from '../shared/base-doc.model';

export interface StreakState extends BaseDoc {
  readonly type: 'streak';
  readonly _id: 'streak:current';
  currentStreak: number;
  longestStreak: number;
  lastActiveDate: string;       // ISO yyyy-mm-dd, local timezone
  graceUsedDate?: string;       // ISO yyyy-mm-dd of last grace day burn
  milestonesReached: number[];  // subset of [3, 7, 30, 100]
  startedAt: string;            // ISO timestamp of first ever streak entry
  updatedAt: string;
}
```

`STORAGE_KEYS` extended with `STREAK_DOC_ID = 'streak:current'`.

### Domain layer

```
core/domain/retention/streak.domain.ts
```

Pure functions, zero Angular deps:

```ts
export type StreakTransition =
  | { kind: 'incremented'; from: number; to: number }
  | { kind: 'grace_used'; streak: number; on: string }
  | { kind: 'reset'; previousStreak: number }
  | { kind: 'milestone_reached'; milestone: 3 | 7 | 30 | 100; streak: number };

export interface StreakEvaluation {
  next: StreakState;
  transitions: StreakTransition[];
}

export function evaluateStreak(
  state: StreakState | null,
  today: string,             // ISO yyyy-mm-dd local
  triggeredByMutation: boolean,
): StreakEvaluation;

export function isMilestone(streak: number): streak is 3 | 7 | 30 | 100;
```

`evaluateStreak` handles three call sites:

1. **Mutation occurred today** (`triggeredByMutation = true`):
   - If `lastActiveDate === today` → no-op.
   - Else compute gap = days between `lastActiveDate` and `today`:
     - gap = 1 → increment.
     - gap = 2 AND grace available → burn grace, increment, set `graceUsedDate = lastActiveDate + 1`.
     - gap >= 2 with no grace, OR gap >= 3 → reset to 1 (today counts), `longestStreak = max(longestStreak, previousStreak)`.
   - After increment, check if `currentStreak` ∈ {3,7,30,100} and not in `milestonesReached`
     → emit `milestone_reached` transition + append to `milestonesReached`.
2. **App resume / cold start** (`triggeredByMutation = false`):
   - If `lastActiveDate === today` → no-op.
   - Else compute gap. If gap >= 2 with no grace → reset.
   - Used to detect "missed yesterday" without requiring a mutation.
3. **First ever entry** (`state === null`): bootstrap with `currentStreak = 0`, `lastActiveDate = ''`. First mutation creates state with `currentStreak = 1`.

Grace availability check: grace available if `graceUsedDate` is null OR
`daysBetween(graceUsedDate, today) >= 7`.

### Service

```
core/services/retention/streak.service.ts
```

Root-scoped Angular service:

```ts
@Injectable({ providedIn: 'root' })
export class StreakService {
  readonly state = signal<StreakState | null>(null);
  readonly currentStreak = computed(() => this.state()?.currentStreak ?? 0);
  readonly longestStreak = computed(() => this.state()?.longestStreak ?? 0);
  readonly milestonesReached = computed(() => this.state()?.milestonesReached ?? []);

  async bootstrap(): Promise<void> { /* load doc, evaluate on resume */ }
  async onMutationLogged(): Promise<void> { /* re-evaluate with triggeredByMutation=true */ }
}
```

#### Wiring

- `HistoryEventManagerService` already persists `PantryEvent` on every mutation.
  Add a hook: emit a lightweight `mutation$` `Subject<void>` (or signal-based effect)
  inside `HistoryEventManagerService` and have `StreakService` subscribe.
- `AppComponent` calls `streakService.bootstrap()` on init AND on
  `appStateChange.isActive === true` (Capacitor App resume).
- Transitions emitted by service:
  - `milestone_reached` → fires `streak_milestone_N` PostHog event + triggers in-app toast (§3) + schedules +24h push (§3).
  - `incremented` → fires `streak_reached` PostHog event with `streak` value.
  - `reset` → fires `streak_broken` PostHog event with `previousStreak`.

## §3 Milestones (in-app + push)

### In-app toast

Triggered immediately when `milestone_reached` transition fires (during the mutation that
crosses 3/7/30/100). Uses Ionic `ToastController`:

- Duration: 4s
- Position: top
- Copy: "¡{{milestone}} días seguidos! 🔥 Sigue así."
- Single-fire per milestone (enforced via `milestonesReached` array on the doc).

### +24h push notification

New notification definition. **Outside** priority registry, following the established v4.5
pattern (welcome id 130, recovery 140-142).

```
core/services/notifications/definitions/streak-milestone.notification.ts
```

- ID: **150**
- Trigger: scheduled at milestone-hit moment, fires `+24h`.
- Copy keys: `notifications.streakMilestone.title`, `notifications.streakMilestone.body`
  - Body: "Llevas {{streak}} días seguidos cuidando tu despensa. ¡Sigue así!"
  - Interpolation uses existing `_named` body convention introduced in v4.5.
- `extra` payload: `{ kind: 'streak_milestone', streak: <number> }`.
- Tap handler: route to `/dashboard`. No item-level deep link required.
- Suppressed when `preferences.notificationsEnabled === false`.
- Suppressed when a higher-priority foreground notification is already pending (priority
  scheduler manages this; id 150 sits outside the priority pool to behave like
  welcome/recovery).

### Scheduler wiring

`StreakService` calls into `NotificationSchedulerService.scheduleStreakMilestone(streak)`
which builds the definition payload and delegates to the existing plugin abstraction.

Cancel-on-open semantics:
- If user opens the app within 24h before push fires → leave push alone (user already
  saw the in-app toast; push acts as a reminder for less-engaged users only). NOT
  cancelled on resume.
- If user reaches the next milestone before previous push fires (e.g. fast streak growth
  on dev day, unlikely) → previous pending milestone push is cancelled and replaced.

### Reset push copy

On `reset` transition with `previousStreak >= 3`:
- Show a one-time **in-app toast** the next time the user opens the app:
  "Tu racha de {{previousStreak}} días terminó. ¡Empieza una nueva hoy!"
- No push. Avoids punitive feel; in-app only.

## Events (PostHog)

| Event | Props | Fires from |
|---|---|---|
| `streak_reached` | `streak` (current value) | `StreakService` on `incremented` transition |
| `streak_milestone_3` | — | on `milestone_reached` with milestone=3 |
| `streak_milestone_7` | — | on `milestone_reached` with milestone=7 |
| `streak_milestone_30` | — | on `milestone_reached` with milestone=30 |
| `streak_milestone_100` | — | on `milestone_reached` with milestone=100 |
| `streak_broken` | `previousStreak` | on `reset` transition |

Separate event names per milestone keep funnels analytable in PostHog without prop
filtering gymnastics.

## Architecture invariants

- Singleton doc (`_id: 'streak:current'`) keeps PouchDB lean. No streak history.
- Longest streak preserved across resets via `longestStreak` field.
- Domain layer purity preserved — `evaluateStreak` is the only place streak math lives.
- `StreakService` is the only Angular-aware piece.
- No backend work.

## Edge cases

| Case | Behavior |
|---|---|
| User crosses midnight in-app (lastActiveDate=yesterday, today changed) | Next mutation evaluates gap=1 → increment correctly |
| User changes device timezone manually | Day boundary may shift; one-time off-by-one possible. Acceptable. Documented in CLAUDE notes. |
| User opens app after 30 days away | `bootstrap()` evaluates gap=30, no grace coverage → `reset`. Reset toast shows on next open. |
| Two mutations same day | First triggers `incremented` (or no-op if already today), second is no-op (`lastActiveDate === today`). |
| Notification permission denied | Milestone push skipped silently. In-app toast still fires. PostHog events still fire. |
| User dismisses milestone toast before reading | One-time per milestone — `milestonesReached` array prevents re-fire. |

## Risks

| Risk | Mitigation |
|---|---|
| EDIT events fire too easily → trivially preserves streak without real engagement | Acceptable for v4.7. Tightening to CONSUME-only deferred until analytics confirm gaming. |
| Streak reset feels punitive even with grace | Reset toast copy non-shaming + grace day reduces frequency. |
| Push fires after user already opened dashboard that day | Acceptable celebration framing; +24h is intentional reminder for less-engaged users. |
| Timezone change inflates / deflates streak by 1 | Edge case acceptable for v4.7. Document in `NOTIFICATIONS.md` / `STATE.md`. |

## Success gates (post-ship, +2 weeks)

- `streak_reached` event volume > 0 (proves the loop runs at all).
- `streak_milestone_3` reaches ≥ 5% of active users (first milestone proves the loop bites).
- D7 retention +1–2 pp (umbrella metric, shared credit with PRO sub-spec).

## Out of scope (this sub-spec)

- Daily check-in toast separate from streak chip
- Weekly recap notification
- Streak leaderboard / social features
- LLM-backed streak copy variations (v4.8 territory)
- Ongoing pinned notification showing streak (deferred)
