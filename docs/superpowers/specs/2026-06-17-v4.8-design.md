## Feature A — CoachMarkService + coach mark post-onboarding

### Problema
Usuario completa onboarding, llega a `/pantry`, no sabe qué hacer. El seed rellena con productos preset que no son suyos — no hay engagement real.

### Solución
Un coach mark de una sola vez sobre el FAB de añadir. Guía al usuario a añadir su primer producto real.

### Servicio: `CoachMarkService`

```
core/services/retention/coach-mark.service.ts
```

API pública:
```typescript
isShown(key: CoachMarkKey): boolean
markShown(key: CoachMarkKey): void
```

Storage: `localStorage` via `LocalStorageService`. No PouchDB — no necesita sync entre dispositivos.

Keys tipadas como union literal:
```typescript
type CoachMarkKey = 'add_first_item' // extensible
```

Diseñado para crecer: añadir un coach mark futuro = una key nueva + un componente. El servicio no cambia.

### Componente: `PantryAddCoachMarkComponent`

```
features/pantry/components/add-coach-mark/
```

- Standalone component, renderizado condicionalmente en `PantryComponent`
- Overlay semitransparente sobre el FAB (`position: fixed`, `z-index` sobre el FAB)
- Texto: usa `pantry.coachMark.addFirstItem` (i18n, 6 idiomas)
- Flecha SVG apuntando al FAB
- Dismiss: tap en overlay O tap en FAB → ambos abren add modal
- Se muestra si: `!coachMark.isShown('add_first_item') && hasSeenOnboarding`

### Condición de display

Condición completa: `hasSeenOnboarding === true` AND `coachMark.isShown('add_first_item') === false`.

No depende de número de items — se muestra una sola vez a cualquier usuario (nuevo o existente que actualice a v4.8). Usuarios con items activos lo descartarán en 1 segundo; usuarios nuevos reciben la guía.

### Analytics

| Evento | Cuándo |
|---|---|
| `coach_mark_shown` | Al renderizar el overlay |
| `coach_mark_dismissed` | Tap en overlay (sin tocar FAB) |
| `coach_mark_tapped` | Tap en FAB desde el coach mark |

### i18n

Añadir en los 6 idiomas (`es`, `en`, `de`, `fr`, `it`, `pt`):

```json
"pantry": {
  "coachMark": {
    "addFirstItem": "Añade tu primer producto real",
    "addFirstItemHint": "Toca el botón para empezar tu despensa"
  }
}
```

## Feature B — Notificaciones proyectadas 7 días

### Problema
`scheduleAll()` agenda 1 notif para mañana. Si el usuario no abre la app, silencio total. Los D2/D5/D10 recovery pushes son genéricos y se eliminan.

### Solución
En cada `scheduleAll()`, proyectar el estado de la despensa para los próximos 7 días y agendar 1 notif por día con el ítem de mayor prioridad para ese día concreto.

### Eliminaciones

- `RecoveryNotificationsService` (`core/services/notifications/recovery-notifications.service.ts`) — **borrar**
- Sus ids 140/141/142 de `NOTIFICATION_IDS` — **borrar**
- Llamada a `recoveryNotif.cancelRecoveryWindow()` en `AppComponent` — **borrar**

### Nueva función de dominio

```
core/domain/notifications/notification.domain.ts
```

```typescript
function projectItemStatusAtDate(item: PantryItem, targetDate: Date): PantryItem
```

Devuelve una copia del item con `expirationStatus` recalculado como si `targetDate` fuera "hoy". No muta el item original. Usa la misma lógica de `pantry-status.domain.ts`.

### Cambio en `NotificationSchedulerService`

```
core/services/notifications/notification-scheduler.service.ts
```

`scheduleAll()` actual: cancela todo → selecciona 1 winner → agenda 1 notif.

`scheduleAll()` nuevo: cancela todo → para cada día 1..7 → proyecta despensa → selecciona winner → agenda notif a las 09:00 hora local de ese día.

```typescript
private async scheduleProjectedNotifications(
  items: PantryItem[],
  now: Date,
  daysAhead: number = 7
): Promise<void>
```

Reglas:
- Hora fija: 09:00 hora local (no configurable en v4.8)
- Si no hay ítem relevante para un día → no agenda ese día (no forzar notif vacía)
- Máximo 7 notifs agendadas en total
- Se reutilizan los builders de contenido existentes — mismos títulos, mismo `extra.itemId`

### Backward compat

`handleNotificationTap` no cambia — ya extrae `extra.itemId` y navega a `/pantry?focusItem=<id>`.

### Analytics

No se añaden nuevos eventos de analytics para esta feature — `notification_scheduled` ya existe y cuenta el total.

## Orden de implementación recomendado

1. `CoachMarkService` + `PantryAddCoachMarkComponent` + i18n (Feature A)

3. ProGuard (opcional, solo si 1+2 verificados en dispositivo)
